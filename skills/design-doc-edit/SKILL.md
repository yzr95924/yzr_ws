# Design Doc Edit

将一个模糊的设计问题或方案，转化为结构清晰、细节充分的 Markdown 设计文档。
产出物为独立的 `.md` 文件，输出路径由用户指定或根据上下文推断，
可供 Code Agent 直接消费以进行编码开发。

## 何时使用 / 不使用

### 使用

- 需要为一个新功能 / 模块撰写详细设计文档
- 需要将口头讨论的设计方案落地为可追溯的 Markdown
- 需要对比多种技术方案并给出推荐
- 需要为已有的骨架设计文档补充细节

### 不使用

- 纯代码实现（直接写代码，不要走设计文档流程）
- 简单的 bug fix / typo 修复
- 已有完整设计文档、只需执行的场景

## 输入 / 输出

### 输入

用户提供的设计问题，可能包含以下信息（不必全部具备）：

- **问题描述**：要解决什么问题
- **约束条件**：性能、兼容性、依赖限制等
- **备选方案**：用户心中已有的思路（可能不完整）
- **关联模块**：与哪些现有模块 / 文档有关联
- **目标路径**：输出文件的位置（可省略，由 skill 根据项目结构推断）

### 输出

一个完整的 Markdown 设计文档。输出路径按以下优先级确定：

1. 用户明确指定的路径
2. 项目中有约定的设计文档目录（如 `doc/`、`docs/design/`、`design/`）
3. 与关联模块同目录

文件名建议采用 `<topic>_design.md` 格式，但应尊重项目已有的命名习惯。

## 章节骨架

设计文档**必须**按以下骨架组织，章节可增减但顺序不可打乱：

```markdown
# <主题名>

## 概述

## 场景分析

## 方案选择

## <核心设计章节>
（根据内容拆分为多个二级章节，每个章节聚焦一个子问题）

## 文件归属
（如涉及新增 / 修改文件）

## 与其他文档的关系
（如有关联文档）
```

### 各章节要求

| 章节 | 是否必须 | 内容要求 |
| --- | --- | --- |
| `# <主题名>` | ✓ | 一级标题，简洁概括设计主题 |
| `## 概述` | ✓ | 1-2 段话说清楚"是什么"和"为什么" |
| `## 场景分析` | ✓ | 拆解问题所在的场景，逐场景分析困难与挑战（详见"场景分析写法"） |
| `## 方案选择` | ✓ | 列出备选方案，对比收益与代价，说明最终选择的理由（详见"方案选择写法"） |
| 核心设计章节 | ✓ | 至少一个；按子问题拆分二级章节，每个章节内聚焦 |
| `## 文件归属` | 推荐 | 表格列出相关文件的路径、用途、是否纳入版本控制 |
| `## 与其他文档的关系` | 可选 | 交叉引用关联文档 |

## 场景分析写法

场景分析是设计文档从"概述"到"核心设计"的桥梁——先把问题空间拆解清楚，
再让后续的设计章节逐一应对。缺少这一步，设计容易直接跳到方案而忽略边界情况。

### 分析流程

```text
用户描述的问题
  │
  ├── 1. 识别场景维度
  │     └── 问题在什么条件下发生？按触发方式、用户角色、数据规模、
  │         环境差异等维度拆出 2-5 个主要场景
  │
  ├── 2. 描述每个场景
  │     ├── 场景名称（简短、可识别）
  │     ├── 触发条件（谁 / 什么时候 / 什么操作）
  │     └── 期望行为（理想情况下应该发生什么）
  │
  ├── 3. 分析困难与挑战
  │     ├── 该场景下"难在哪里"（技术约束、状态复杂度、边界条件）
  │     ├── 是否有已知的失败模式
  │     └── 与其他场景是否存在冲突或耦合
  │
  └── 4. 汇总为场景矩阵
        └── 用表格呈现全貌，为后续设计章节提供索引
```

### 场景识别原则

- **按触发方式拆**：用户主动触发 vs 系统自动触发 vs 外部事件驱动
- **按用户角色拆**：不同角色对同一功能的期望和约束可能不同
- **按数据规模 / 复杂度拆**：简单 case vs 复杂 case 往往需要不同的处理策略
- **按异常路径拆**：正常流程 vs 异常 / 降级 / 重试等场景
- **按环境差异拆**：不同运行环境、平台、配置下的行为差异

不必穷举所有维度——选择与当前问题最相关的 2-3 个维度来拆分，
最终产出 2-5 个主要场景。场景太少则分析不充分，太多则说明粒度需要合并。

### 输出格式

场景分析章节使用三级子章节 + 汇总矩阵的组合：

```markdown
## 场景分析

### 场景一：<场景名称>

**触发条件**：<谁 / 什么时候 / 什么操作>

**期望行为**：<理想情况>

**困难与挑战**：

- <挑战 1>：<为什么难，技术约束是什么>
- <挑战 2>：<边界条件 / 失败模式>

### 场景二：<场景名称>

（同上结构）

### 场景汇总

| 场景 | 触发条件 | 核心挑战 | 涉及的设计章节 |
| --- | --- | --- | --- |
| 场景一 | ... | ... | §<章节名> |
| 场景二 | ... | ... | §<章节名> |
```

### 质量要求

1. **场景必须可辨识**：读者看完场景名称和触发条件，能立刻对应到实际使用中的某个情况
2. **挑战必须具体**：写"难在 X 因为 Y"，不要写"这个场景比较复杂"
3. **场景之间要有区分度**：如果两个场景的挑战完全相同，说明它们应该合并
4. **挑战要能映射到设计章节**：汇总矩阵的"涉及的设计章节"列不能留空——每个挑战都应该有对应的设计章节去解决它；如果某个挑战没有对应章节，说明文档缺少设计内容

## 方案选择写法

方案选择是设计文档的决策中枢——在场景分析明确"问题是什么"之后，
回答"用什么方案解决"以及"为什么选它而不选别的"。缺少这一步，
读者（包括未来的自己和 Code Agent）无法理解设计决策的来龙去脉，
容易在后续迭代中重复论证或误改方向。

### 方案评估流程

```text
场景分析产出的挑战列表
  │
  ├── 1. 提出备选方案
  │     ├── 列出 2-4 个可行的技术方案（含"不处理"作为基线）
  │     └── 每个方案用一句话概括核心思路
  │
  ├── 2. 逐方案评估
  │     ├── 收益：解决哪些场景的挑战，带来什么额外价值
  │     ├── 代价：实现复杂度、性能开销、维护成本、兼容性风险
  │     └── 约束适配：是否满足项目已有的技术 / 架构 / 资源约束
  │
  ├── 3. 对比与推荐
  │     ├── 用对比矩阵呈现关键维度上的差异
  │     ├── 明确推荐方案
  │     └── 说明推荐理由（收益 > 代价 的具体论据）
  │
  └── 4. 记录否决理由
        └── 每个被否决方案，说明"为什么不选"（一句话）
```

### 方案提出原则

- **至少 2 个备选方案**：如果只有 1 个方案，说明分析不充分；强制自己找替代方案
- **包含"不处理"基线**：明确"如果什么都不做会怎样"，帮助判断方案是否真的值得投入
- **方案要可行**：每个方案都应该是技术上可落地的，不要放一个明显不靠谱的方案来凑数
- **粒度一致**：备选方案之间应该是同一层面的选择（不要拿"用 Redis"和"重写整个系统"对比）

### 方案选择输出格式

```markdown
## 方案选择

### 备选方案

**方案 A：<一句话概括>**

<2-3 句话描述核心思路>

- 收益：<解决了什么，带来什么价值>
- 代价：<实现复杂度、风险、限制>

**方案 B：<一句话概括>**

（同上结构）

**方案 C：不处理（基线）**

维持现状，不做改动。适用于：<什么时候"不做"也是合理选择>

### 对比矩阵

| 维度 | 方案 A | 方案 B | 不处理 |
| --- | --- | --- | --- |
| 场景覆盖 | ... | ... | — |
| 实现复杂度 | ... | ... | — |
| 维护成本 | ... | ... | — |
| 扩展性 | ... | ... | — |

### 推荐方案

**推荐：方案 A**

理由：

1. <收益论据 1>
2. <收益论据 2>
3. <与场景分析中挑战的对应关系>

### 否决理由

- **方案 B**：<一句话说明为什么不选>
- **不处理**：<一句话说明为什么不能维持现状>
```

### 方案选择质量要求

1. **收益必须对应场景**：每个收益点应能追溯到场景分析中的具体挑战，不要写"提升用户体验"这类空话
2. **代价必须诚实**：不要回避方案的缺点——如果实现复杂就写复杂，如果有性能开销就写开销
3. **否决理由必须具体**：写"方案 B 不支持 X 场景的 Y 需求"，不要写"方案 B 不够好"
4. **推荐方案必须有论据**：不能只说"推荐方案 A"，要解释为什么 A 的收益大于其代价
5. **对比维度要有区分度**：如果所有方案在某个维度上评价相同，这个维度不值得列

## 视觉规范

以下视觉元素是设计文档的核心表达手段，按需使用。

### ASCII 架构图

用于表达组件关系、模块层次。使用 `text` 代码块：

```text
调度层
  ├── engine/
  │     ├── base.py          # 抽象基类
  │     └── adapter_a.py     # 适配器 A
  └── manager.py             # 管理模块
```

### 状态机 / 流程图

使用 box-drawing 字符（`┌ ┐ └ ┘ │ ─ ►`）绘制状态转换和流程：

```text
(new) ──► active ──► paused ──► completed
           ▲          │
           └──────────┘
           (resume)
```

### 目录树

用 `text` 代码块 + 缩进表示文件 / 目录结构：

```text
module/
├── config.json      # 配置文件
├── core.py          # 核心逻辑
└── tests/           # 测试目录
```

### 表格

Markdown 表格用于结构化对比和属性列举：

```markdown
| 能力 | 方案 A | 方案 B |
| --- | --- | --- |
| 恢复会话 | 支持 | 不支持 |
| 导出记录 | ✗ | ✓ |
```

### 代码块

- **目标语言**：用于接口定义、伪代码、关键实现片段；包含 docstring 和注释
- **JSON / JSONC**：用于数据 schema，带行内 `//` 注释说明字段含义
- **Shell**：用于命令示例和用户交互流程

代码块的语言标注应与项目实际使用的语言一致。注释语言跟随项目的注释规范。

## 硬性内容规则

1. **概述先行**：第一个二级章节必须是 `## 概述`，用 1-2 段话说清背景与动机
2. **不虚构模块**：只引用项目中已存在或本文档计划创建的模块，不凭空编造
3. **文件归属表**：如果文档涉及新增 / 修改文件，必须包含文件归属表，标明路径、用途、是否纳入版本控制
4. **交叉引用**：引用其他文档时使用相对路径链接
5. **行宽约束**：正文单行不超过 120 字符（代码块与表格行除外）
6. **尊重项目风格**：如果项目有 CLAUDE.md 或其他规范，文档风格（注释语言、中英文空格等）必须与之保持一致
7. **表格列序**：多方案对比时，按"能力 / 属性 | 方案 A | 方案 B"组织

## 工作流

```text
用户提出设计问题
  │
  ├── 1. 理解上下文
  │     ├── 阅读用户描述
  │     ├── 如果描述模糊，向用户提问澄清（不超过 3 个问题）
  │     ├── 阅读项目的 CLAUDE.md / README.md，了解项目规范
  │     └── 阅读关联的已有设计文档，理解现有设计上下文
  │
  ├── 2. 场景分析
  │     ├── 按触发方式 / 用户角色 / 数据规模等维度拆解场景（2-5 个）
  │     ├── 逐场景描述触发条件、期望行为、困难与挑战
  │     ├── 与用户确认场景拆解是否完整、挑战是否准确
  │     └── 输出场景汇总矩阵（草稿）
  │
  ├── 3. 方案选择
  │     ├── 提出 2-4 个备选方案（含"不处理"基线）
  │     ├── 逐方案评估收益与代价
  │     ├── 构建对比矩阵，明确推荐方案与否决理由
  │     └── 与用户确认方案选择
  │
  ├── 4. 确定文档结构
  │     ├── 拟定一级标题
  │     ├── 确定目标文件路径（用户指定 / 项目约定 / 推断）
  │     ├── 根据场景分析 + 方案选择拆出核心设计章节
  │     │     （每个挑战映射到至少一个章节，方案的关键决策也要有对应章节）
  │     └── 判断是否需要 文件归属 / 与其他文档的关系 章节
  │
  ├── 5. 填充内容
  │     ├── 撰写概述（1-2 段话）
  │     ├── 撰写场景分析章节（三级子章节 + 汇总矩阵）
  │     ├── 撰写方案选择章节（备选方案 + 对比矩阵 + 推荐与理由）
  │     ├── 逐个填充核心设计章节
  │     │     ├── 选择合适的视觉元素（架构图 / 表格 / 代码块）
  │     │     └── 确保代码示例符合项目注释规范
  │     ├── 补充文件归属表（如有新增文件）
  │     └── 补充交叉引用（如有关联文档）
  │
  ├── 6. 自检
  │     ├── 概述是否说清了"是什么"和"为什么"
  │     ├── 场景分析是否充分（场景可辨识、挑战具体、映射到设计章节）
  │     ├── 方案选择是否有说服力（备选方案可行、收益对应场景、否决理由具体）
  │     ├── 章节顺序是否符合骨架
  │     ├── 文件归属表是否完整（如适用）
  │     └── 风格是否与项目已有文档一致
  │
  └── 7. 输出
        ├── 如果目标文件已存在：与用户确认后覆盖或追加
        └── 如果目标文件不存在：新建文件并写入
```

### 已有骨架文档时的处理

如果目标文件已存在但内容很少（如只有标题），视为"骨架文档"：

1. 保留已有的标题和章节结构
2. 在已有章节下补充详细内容
3. 如有必要新增章节，追加在已有章节之后
4. 写入前与用户确认补充方案
